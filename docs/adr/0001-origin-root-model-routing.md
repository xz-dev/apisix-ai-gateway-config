# Origin/root model routing

We route direct provider-origin calls with `origin/<provider>/<raw-provider-model-id>` and reserve root model IDs for explicit model resolution rules. Provider deployments such as `ollama-1` and `ollama-2` are hidden behind a logical provider and load-balanced equally by default, while root fallback rules are expanded by the config renderer into explicit APISIX `ai-proxy-multi` routes that reference origin targets; this avoids accidental cross-provider fallback while still allowing approved root-model fallback such as Ollama DeepSeek models falling back to official DeepSeek on 429/5xx. We intentionally do not support runtime regex capture inside APISIX for model routing because `ai-proxy-multi` instances use static `options.model` values. Model capabilities, especially reasoning support and effort values, are treated as model-centric metadata: a provider catalog that omits reasoning flags must not be interpreted as proof that the underlying model lacks reasoning.
